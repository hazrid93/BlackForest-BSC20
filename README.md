## Key Features & Highlights

BlackForest (BFT) is a BEP-20 token on Binance Smart Chain that rewrites the SafeMoon/Rfi model into a clean, modular, multi-fee architecture. Its standout trait is a fully data-driven fee engine: instead of hardcoding each tax like SafeMoon does, fees are defined as array entries and dispatched generically ŌĆö letting a deployer add buy/sell taxes, wallets, and destinations without touching the transfer logic. The contract is authored by "0xSaiya" and targets Solidity 0.8.4 with optimization.

- **Rfi dual-balance reflection rewards** ŌĆö Holders earn a share of every transaction fee without any claim action or iteration over holders. The contract keeps two parallel mappings (`_reflectedBalances` / `_balances`) and a single `_reflectedSupply` initialized to `~uint256(0) - (~uint256(0) % TOTAL_SUPPLY)`. The clever bit is in `_redistribute` (line 597): instead of crediting each holder, it *reduces* `_reflectedSupply` by `rFee`, which raises the `_getCurrentRate()` ratio (`rSupply / tSupply`). This makes every non-excluded holder's `balanceOf` automatically increase ŌĆö a constant-time reward distribution that scales to unlimited holders with no gas-per-holder cost.

- **Data-driven fee engine with buy/sell asymmetry** ŌĆö All taxes are registered via `_addFee(FeeType name, uint256 value, address recipient, bool isBuy)` into a `Fee[]` array (line 159). Six fee types exist: `Rfi`, `Burn`, `Liquidity`, `External`, and `ExternalToETH`. The contract defines **different rates for buys vs. sells** ŌĆö e.g. Rfi 2.5% buy / 4% sell, Burn 1.5% buy / 3% sell, and a flat 4% dev tip both ways ŌĆö totaling 10.5% on buys and 15% on sells. `_takeTransactionFees` (line 868) detects direction via `sender == _pair` and skips fees whose `isBuy` flag doesn't match, so adding a new tax is a one-line `_addFee` call rather than a logic rewrite. The author's header notes that fees are in per-mille (based on `FEES_DIVISOR = 10**3`) and warns that PancakeSwap's 49.9% max slippage means fees over ~35% will fail.

- **Auto-liquidity "swap and liquify" with Certik-audit fixes** ŌĆö The `Liquifier` contract (line 611) accumulates the liquidity-fee tokens at `address(this)`, and when the balance crosses `numberOfTokensToSwapToLiquidity` (= `TOTAL_SUPPLY / 1000`, 0.1%), `_beforeTokenTransfer` (line 859) triggers `liquify`. `_swapAndLiquify` splits the balance in half, swaps half for BNB via `swapExactTokensForETHSupportingFeeOnTransferTokens`, then pairs the rest with the BNB through `addLiquidityETH`. Notably it **fixes two known SafeMoon/Certik flaws**: the `withdrawLockedEth` function (line 800) lets the manager drain residual BNB that would otherwise be locked forever (Certik SSL-03), and LP tokens are minted to `owner()` (line 754) with the residual-BNB issue documented inline.

- **Reentrancy-guarded swap via `lockTheSwap` modifier** ŌĆö Because `swapAndLiquify` calls back into the PancakeSwap router (which could re-enter the token's transfer and recursively trigger another liquify), the contract guards it with a `modifier lockTheSwap` (line 636) setting `inSwapAndLiquify = true` for the duration. `liquify` (line 663) additionally checks `!inSwapAndLiquify && sender != _pair` before firing, preventing both reentrancy and circular liquidity events during buys. The `withdrawLockedEth` function applies the same pattern: it zeros `withdrawableBalance` *before* the external `transfer` (line 805ŌĆō807), a classic checks-effects-interactions guard.

- **Anti-whale transaction & wallet caps** ŌĆö Hard limits enforced inside `_transfer` (line 454): `maxTransactionAmount = TOTAL_SUPPLY / 100` (1%) caps any single transfer, and `maxWalletBalance = TOTAL_SUPPLY / 50` (2%) caps any recipient's holding. The owner and burn address are whitelisted as unlimited senders/recipients via `_isUnlimitedSender`/`_isUnlimitedRecipient` (lines 441ŌĆō452), and the PancakeSwap pair is excluded from the wallet-cap check (line 481) so sells back to the pool aren't blocked. An `Antiwhale` contract (line 819) exposes `_getAntiwhaleFees(sendersBalance, amount)` as a documented hook that currently returns the flat `sumOfFees` but is explicitly designed to scale fees by wallet size.

- **Dual access control: Owner + Manager separation** ŌĆö Beyond standard `Ownable` (with `onlyOwner`, `renounceOwnership`, and a timelocked `lock`/`unlock` ownership-transfer pattern in the imports at lines 106ŌĆō117), the contract adds a distinct `Manageable` role (line 119) gated by `onlyManager`. Operational parameters that are safer to rotate are delegated to the manager ŌĆö `setPreseableEnabled`, `setRouterAddress`, `setSwapAndLiquifyEnabled`, and `withdrawLockedEth` ŌĆö while ownership controls (`excludeFromReward`, `setExcludedFromFee`) stay with the owner. This separation lets a team rotate the manager key for day-to-day ops without exposing ownership-renouncement power.

- **Presale mode that disables fees and safety limits** ŌĆö A `Presaleable` contract (line 230) flips an `isInPresale` flag via `onlyManager`. When true, `_transfer` skips the max-tx/max-wallet checks (line 463) and `_takeFees` returns early (line 528), so presale distributions and initial liquidity provisioning happen tax-free and uncapped. Once presale ends, the manager flips the flag and full fee/anti-whale enforcement kicks in ŌĆö a clean migration mechanism for launch.

- **"Soft" burn that boosts redistribution instead of shrinking supply** ŌĆö The `burn` function (line 324) sends tokens to a vanity burn address `0x0000000000000000000000000000000299792458` (the speed of light in m/s, per the author's comment) rather than reducing `totalSupply`. Because the burn address is excluded from rewards in the constructor (`_exclude(burnAddress)`, line 847), burned tokens are removed from `_getCurrentSupply`'s `tSupply` (line 562), which *increases* the redistribution rate to remaining holders. The contract also hard-blocks transfers *from* the burn address (line 457) and burns from the burn address (line 328) to prevent resurrection.

- **Reward-exclusion bookkeeping with safe supply reconciliation** ŌĆö To keep the reflection math exact, excluded accounts (owner, contract, pair, burn) track their true balance in `_balances` while everyone else uses reflections. `_getCurrentSupply` (line 550) iterates the `_excluded[]` array and subtracts each excluded balance from both `rSupply` and `tSupply`. It includes a defensive invariant check (line 560): if any excluded reflected balance exceeds `rSupply`, it resets to the full `(_reflectedSupply, TOTAL_SUPPLY)` ŌĆö preventing a catastrophic division-by-zero or negative-rate state that would otherwise brick all `balanceOf` reads. `includeInReward`/`excludeFromReward` let the owner migrate accounts between the two systems, with the classic swap-and-pop array removal (lines 417ŌĆō425).

- **Environment-aware PancakeSwap routing** ŌĆö `Liquifier` ships with three hardcoded router addresses (lines 621ŌĆō625): PancakeSwap V1 mainnet `0x05fF2B0DŌĆ”`, V2 mainnet `0x10ED43C7ŌĆ”`, and testnet `0xD99D1c33ŌĆ”`, selected by an `Env` enum (`Testnet`, `MainnetV1`, `MainnetV2`) passed to `initializeLiquiditySwapper`. `_setRouterAddress` (line 687) instantiates the router and calls `createPair(address(this), WETH)` on its factory, so the LP pair is created at construction. The `BlackForest` constructor deploys with `Env.Testnet` and pre-approves the router for `~uint256(0)` (line 960) to allow unlimited swap-and-liquify approvals without per-call allowance management.
